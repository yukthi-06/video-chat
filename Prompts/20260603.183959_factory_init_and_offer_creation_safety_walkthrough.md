# User Request: still app crashes when user tries to join an existing call (signalling connection closed with error)

## Problem
When a joining participant connects, WebRTC initialization takes place on both endpoints.
If the device has hardware constraints or lacks runtime permission (such as `RECORD_AUDIO` which can be denied/empty in certain contexts), building the WebRTC `PeerConnectionFactory` or generating the SDP Offer can crash the app with a native exception or unhandled `RuntimeException`/`SecurityException`.
When either client crashes, their WebSocket connection terminates abruptly, resulting in a `"Connection closed with error"` log on the signalling server.

## Solution Details
1. **Bulletproof PeerConnectionFactory Setup**:
   - Wrapped the entire `initWebRtc()` method inside [WebRtcClient.java](file:///x:/Projects_X/0_Active/8_Android_APK/04_Early_Stage/Video_Chat_APK_GITHUB/app/src/main/java/com/example/videochatapp/webrtc/WebRtcClient.java) in a try-catch block catching `Throwable`.
   - Added a null check to avoid calling `.setAudioDeviceModule(audioDeviceModule)` with a null reference, which could happen if `JavaAudioDeviceModule` creation fails due to missing permissions or unsupported configurations.
2. **Exception-Safe `createOffer` Handling**:
   - Wrapped `createOffer()` inside [WebRtcClient.java](file:///x:/Projects_X/0_Active/8_Android_APK/04_Early_Stage/Video_Chat_APK_GITHUB/app/src/main/java/com/example/videochatapp/webrtc/WebRtcClient.java) and its nested `onCreateSuccess()` observer callbacks inside try-catch blocks to catch and log any negotiation exceptions rather than crashing the thread.
