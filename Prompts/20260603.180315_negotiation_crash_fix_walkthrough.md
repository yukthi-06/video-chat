# User Request: app crashes when user tries to join an existing call

## Problem
When a joining user connects to an existing call, the negotiation sequence is triggered:
1. They receive an SDP Offer from the creator.
2. WebRTC PeerConnection handles the remote offer and attempts to create an Answer.
3. If there is a codec mismatch, invalid state, or thread synchronization issues inside `handleOffer()`, `handleAnswer()`, or `addIceCandidate()`, WebRTC raises native errors or runtime exceptions.
4. Because these calls were not individually isolated in try-catch blocks on both the UI thread (`CallActivity.java`) and background threads (`WebRtcClient.java`), any exceptions caused the app to crash immediately.

## Solution Details
1. **Isolated SDP Handlers**:
   - Wrapped `handleOffer()`, `handleAnswer()`, and `addIceCandidate()` inside [WebRtcClient.java](file:///x:/Projects_X/0_Active/8_Android_APK/04_Early_Stage/Video_Chat_APK_GITHUB/app/src/main/java/com/example/videochatapp/webrtc/WebRtcClient.java) in try-catch blocks catching `Throwable`.
   - Isolated the internal callback success (`onCreateSuccess()`) when creating local descriptions/answers, logging errors instead of crashing the thread.
2. **UI Thread Thread-Safety**:
   - Wrapped [CallActivity.java](file:///x:/Projects_X/0_Active/8_Android_APK/04_Early_Stage/Video_Chat_APK_GITHUB/app/src/main/java/com/example/videochatapp/activities/CallActivity.java)'s callback delegates (`onOfferReceived()`, `onAnswerReceived()`, and `onIceCandidateReceived()`) inside try-catch blocks on the UI thread to protect the application lifecycle.
