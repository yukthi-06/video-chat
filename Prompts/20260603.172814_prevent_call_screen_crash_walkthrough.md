# User Request: on clicking "start a call", a call screen appears adnd immediately app crashes

## Problem
When initializing `WebRtcVideoRecorder`, it attempts to create and bind the EglBase context on the device using `EglBase.CONFIG_RECORDABLE`. On some Android devices and emulators, EGL configurations for recordable surfaces are not supported by the hardware, or mismatch with the shared context configuration.
When this happens, WebRTC's EGL helper throws an `AssertionError` (which is a subclass of `Error`, not `Exception`).
Because the `start()` method in `WebRtcVideoRecorder.java` previously caught only `IOException` (and then `Exception` in a subsequent minor edit), the `AssertionError` bubbled up uncaught, causing the app to crash immediately upon entering `CallActivity`.

## Solution Details
1. **Robust EglBase Creation Fallback**:
   Updated [WebRtcVideoRecorder.java](file:///x:/Projects_X/0_Active/8_Android_APK/04_Early_Stage/Video_Chat_APK_GITHUB/app/src/main/java/com/example/videochatapp/webrtc/WebRtcVideoRecorder.java) to wrap `EglBase.create(sharedContext, EglBase.CONFIG_RECORDABLE)` in a try-catch block. If it fails (due to unsupported configurations), it falls back to `EglBase.create(sharedContext, EglBase.CONFIG_PLAIN)`.
2. **Catch All Throwables**:
   Changed the catch-blocks in `start()`, `onFrame()`, and `drainEncoder()` to catch `Throwable` instead of `Exception`. This guarantees that even if WebRTC throws native `AssertionError`s, EGL errors, or native runtime issues during encoding setup/run, they are safely caught, logged, and handled without crashing the app.
