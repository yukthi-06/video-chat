# User Request: app still crashes at start of call

## Problem
In `WebRtcClient.java`, the `DefaultVideoEncoderFactory` and `DefaultVideoDecoderFactory` were being initialized with a `null` EglBase context parameter:
```java
DefaultVideoEncoderFactory defaultVideoEncoderFactory = new DefaultVideoEncoderFactory(null, true, true);
```
Setting the EGL context to `null` while requesting hardware acceleration (`true, true`) causes native crashes or `NullPointerException`s when starting capture/encoding under modern WebRTC android libraries, because hardware-accelerated video codecs require a valid shared EGL context to bind input/output frame textures.

## Solution Details
1. **Pass Shared EglBase Context**:
   Updated the constructor of [WebRtcClient.java](file:///x:/Projects_X/0_Active/8_Android_APK/04_Early_Stage/Video_Chat_APK_GITHUB/app/src/main/java/com/example/videochatapp/webrtc/WebRtcClient.java) to accept a shared `EglBase.Context` object, and propagated it to both `DefaultVideoEncoderFactory` and `DefaultVideoDecoderFactory`.
2. **Instantiate Client with Context**:
   Updated [CallActivity.java](file:///x:/Projects_X/0_Active/8_Android_APK/04_Early_Stage/Video_Chat_APK_GITHUB/app/src/main/java/com/example/videochatapp/activities/CallActivity.java) to pass the initialized rendering `eglBase.getEglBaseContext()` context into the `WebRtcClient` constructor. This aligns the camera capturer, the view renderers, and the video encoders under the same shared OpenGL EGL environment, preventing native/hardware texture mapping crashes.
