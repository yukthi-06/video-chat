# Walkthrough - Remote Audio Track Recording Limitation Investigation

We investigated fixing the recording of the **Remote Audio Track** by attempting to capture the raw remote audio frames at the Java/Android layer.

## Findings and Technical Limitations

1. **No Java `AudioSink` bindings**:
   We attempted to attach an `AudioSink` to the remote `AudioTrack` inside `WebRtcClient.onAddTrack()` using:
   ```java
   audioTrack.addSink(new AudioSink() { ... });
   ```
   However, the compiler returned:
   ```
   error: cannot find symbol: class AudioSink
   ```
   This is because the `im.conversations.webrtc:webrtc-android:119.0.0` library build (which is the standard WebRTC Android SDK) does **not** expose the `AudioSink` JNI interface or the `addSink` method on `AudioTrack` in its public Java API bindings.

2. **No `JavaAudioDeviceModule` Playout callbacks**:
   The standard `JavaAudioDeviceModule` only provides `SamplesReadyCallback` for the microphone recording (input) side via `onWebRtcAudioRecordSamplesReady`. It does not expose a callback for playout (output/remote peer speaker audio) raw samples.

## Resolution
To ensure the app remains fully functional and builds correctly, we have reverted all changes in:

As a result, the code is back to a clean, compile-ready state. Because the WebRTC Android library does not expose these audio playout hooks to Java, remote audio recording is technically impossible without rebuilding the entire native C++ library (`libjingle_peerconnection_so.so`) from scratch to inject a JNI bridge for playout data.
