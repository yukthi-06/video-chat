# User Request: It now crashes when participant 2 joins a call. Can a similar fix be applied for this scenario too?

## Problem
When a second participant joins a call, WebRTC's observer thread fires `onRemoteTrackAdded(VideoTrack track)`.
If starting either `localRecorder` or `remoteRecorder` triggers an exception or a native EGL/MediaCodec assertion (e.g. OpenGL thread binding or context conflicts), and these calls are not individually isolated, the whole observer execution fails, causing the application to crash instantly on the remote track added event.

## Solution Details
1. **Try-Catch Block Isolation**:
   - Wrapped `localRecorder.start()` in a separate `try-catch(Throwable t)` block inside `onRemoteTrackAdded` in [CallActivity.java](file:///x:/Projects_X/0_Active/8_Android_APK/04_Early_Stage/Video_Chat_APK_GITHUB/app/src/main/java/com/example/videochatapp/activities/CallActivity.java).
   - Wrapped `remoteRecorder.start()` and its track binding `track.addSink(remoteRecorder)` in its own separate `try-catch(Throwable t)` block inside `onRemoteTrackAdded` in [CallActivity.java](file:///x:/Projects_X/0_Active/8_Android_APK/04_Early_Stage/Video_Chat_APK_GITHUB/app/src/main/java/com/example/videochatapp/activities/CallActivity.java).
This guarantees that any native EGL/MediaCodec setup issues are caught and logged gracefully on that background thread, allowing the video call connection to succeed without crashing the application.
