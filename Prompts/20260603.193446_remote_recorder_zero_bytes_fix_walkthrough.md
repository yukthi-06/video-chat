# User Request: only local recording is saved, remote recording has created a mp4 of zero bytes.

## Problem
During the initialization of `localRecorder` and `remoteRecorder` inside `CallActivity`, the `start()` method was called on the starting thread (e.g. WebRTC signaling observer thread).
In `start()`, `recorderEglBase.makeCurrent()` was called to configure the MediaCodec encoder surfaces.
However, because EGL contexts can only be active on a **single thread at a time**, and `start()` did not unbind/detach the EGL context after it finished setting up the recorder parameters:
1. The recorder's EGL context remained bound to the starting thread.
2. When the remote `onFrame()` callback was later executed on the WebRTC decoding thread, calling `recorderEglBase.makeCurrent()` failed with `EGL_BAD_ACCESS` (since the context was still current on the starting thread).
3. As a result, the remote recorder could not clear, draw, or swap remote frames, leading to a zero-byte `.mp4` file.

## Solution Details
- **Release EGL Context on Startup**:
  Modified `start()` inside [WebRtcVideoRecorder.java](file:///x:/Projects_X/0_Active/8_Android_APK/04_Early_Stage/Video_Chat_APK_GITHUB/app/src/main/java/com/example/videochatapp/webrtc/WebRtcVideoRecorder.java) to save the starting thread's EGL context before making any modifications.
  Added a `finally` block to restore/detach the starting thread's EGL context before `start()` returns. This frees up the context so the decoding thread can successfully call `makeCurrent()` inside `onFrame()` and write the remote video feed to disk.
